<span data-ttu-id="60166-104">Большинство современных веб-приложений предоставляют API, которые клиенты могут использовать для взаимодействия с приложением.</span><span class="sxs-lookup"><span data-stu-id="60166-104">Most modern web applications expose APIs that clients can use to interact with the application.</span></span> <span data-ttu-id="60166-105">Качественно спроектированный API должен поддерживать следующее:</span><span class="sxs-lookup"><span data-stu-id="60166-105">A well-designed web API should aim to support:</span></span>

## <a name="introduction-to-rest"></a><span data-ttu-id="60166-114">Общие сведения о REST</span><span class="sxs-lookup"><span data-stu-id="60166-114">Introduction to REST</span></span>

<span data-ttu-id="60166-115">В 2000 году Рой Филдинг предложил архитектурный подход к разработке веб-служб — передачу репрезентативного состояния (REST).</span><span class="sxs-lookup"><span data-stu-id="60166-115">In 2000, Roy Fielding proposed Representational State Transfer (REST) as an architectural approach to designing web services.</span></span> <span data-ttu-id="60166-116">REST — это архитектурная концепция создания распределенных систем на основе гиперсред.</span><span class="sxs-lookup"><span data-stu-id="60166-116">REST is an architectural style for building distributed systems based on hypermedia.</span></span> <span data-ttu-id="60166-117">Модель REST не зависит от каких-либо базовых протоколов и не требует привязки к HTTP.</span><span class="sxs-lookup"><span data-stu-id="60166-117">REST is independent of any underlying protocol and is not necessarily tied to HTTP.</span></span> <span data-ttu-id="60166-118">Однако наиболее распространенные реализации REST используют HTTP в качестве протокола приложения. Поэтому это руководство посвящено разработке REST API для HTTP.</span><span class="sxs-lookup"><span data-stu-id="60166-118">However, most common REST implementations use HTTP as the application protocol, and this guide focuses on designing REST APIs for HTTP.</span></span>

<span data-ttu-id="60166-119">Основное преимущество при использовании REST с протоколом HTTP заключается в применении открытых стандартов и отсутствии необходимости в определенной реализации API или клиентских приложений.</span><span class="sxs-lookup"><span data-stu-id="60166-119">A primary advantage of REST over HTTP is that it uses open standards, and does not bind the implementation of the API or the client applications to any specific implementation.</span></span> <span data-ttu-id="60166-120">Например, веб-службу REST можно написать в ASP.NET, а клиентские приложения могут использовать любой язык или набор инструментов, позволяющие создавать HTTP-запросы и анализировать HTTP-ответы.</span><span class="sxs-lookup"><span data-stu-id="60166-120">For example, a REST web service could be written in ASP.NET, and client applications can use any language or toolset that can generate HTTP requests and parse HTTP responses.</span></span>

<span data-ttu-id="60166-121">Ниже приведены некоторые основные принципы проектирования интерфейсов RESTful API, использующих протокол HTTP:</span><span class="sxs-lookup"><span data-stu-id="60166-121">Here are some of the main design principles of RESTful APIs using HTTP:</span></span>

- <span data-ttu-id="60166-122">REST API разрабатываются на основе *ресурса*, который может быть любым типом объекта, данных или службы, к которому может получить доступ клиент.</span><span class="sxs-lookup"><span data-stu-id="60166-122">REST APIs are designed around *resources*, which are any kind of object, data, or service that can be accessed by the client.</span></span>

- <span data-ttu-id="60166-123">У ресурса есть *идентификатор* (универсальный код ресурса (URI)), который уникально идентифицирует этот ресурс.</span><span class="sxs-lookup"><span data-stu-id="60166-123">A resource has an *identifier*, which is a URI that uniquely identifies that resource.</span></span> <span data-ttu-id="60166-124">Например, URI для определенного клиентского заказа может быть таким:</span><span class="sxs-lookup"><span data-stu-id="60166-124">For example, the URI for a particular customer order might be:</span></span>

    ```HTTP
    https://adventure-works.com/orders/1
    ```

- <span data-ttu-id="60166-125">Клиенты взаимодействуют со службой путем обмена *представлениями* ресурсов.</span><span class="sxs-lookup"><span data-stu-id="60166-125">Clients interact with a service by exchanging *representations* of resources.</span></span> <span data-ttu-id="60166-126">Многие веб-API используют JSON в качества формата для обмена.</span><span class="sxs-lookup"><span data-stu-id="60166-126">Many web APIs use JSON as the exchange format.</span></span> <span data-ttu-id="60166-127">Например, в результате запроса GET к приведенному выше URI может вернуться такой текст ответа:</span><span class="sxs-lookup"><span data-stu-id="60166-127">For example, a GET request to the URI listed above might return this response body:</span></span>

    ```json
    {"orderId":1,"orderValue":99.90,"productId":1,"quantity":1}
    ```

- <span data-ttu-id="60166-128">Интерфейсы REST API используют единый интерфейс, который позволяет отделить реализации клиента и службы.</span><span class="sxs-lookup"><span data-stu-id="60166-128">REST APIs use a uniform interface, which helps to decouple the client and service implementations.</span></span> <span data-ttu-id="60166-129">Для REST API, созданных на основе протокола HTTP, единый интерфейс будет использовать стандартные HTTP-команды для выполнения операций с ресурсами.</span><span class="sxs-lookup"><span data-stu-id="60166-129">For REST APIs built on HTTP, the uniform interface includes using standard HTTP verbs to perform operations on resources.</span></span> <span data-ttu-id="60166-130">Наиболее часто выполняемые операции: GET, POST, PUT, PATCH и DELETE.</span><span class="sxs-lookup"><span data-stu-id="60166-130">The most common operations are GET, POST, PUT, PATCH, and DELETE.</span></span>

- <span data-ttu-id="60166-131">REST API использует модель запросов без отслеживания состояния.</span><span class="sxs-lookup"><span data-stu-id="60166-131">REST APIs use a stateless request model.</span></span> <span data-ttu-id="60166-132">HTTP-запросы должны быть независимыми и могут создаваться в любом порядке, поэтому сохранение сведений о переходном состоянии между запросами не представляется возможным.</span><span class="sxs-lookup"><span data-stu-id="60166-132">HTTP requests should be independent and may occur in any order, so keeping transient state information between requests is not feasible.</span></span> <span data-ttu-id="60166-133">Сведения хранятся только в самих ресурсах, и каждый запрос должен быть атомарной операцией.</span><span class="sxs-lookup"><span data-stu-id="60166-133">The only place where information is stored is in the resources themselves, and each request should be an atomic operation.</span></span> <span data-ttu-id="60166-134">Благодаря этому ограничению веб-службы имеют высокую масштабируемость, так как нет необходимости сохранять сходство между клиентами и конкретными серверами.</span><span class="sxs-lookup"><span data-stu-id="60166-134">This constraint enables web services to be highly scalable, because there is no need to retain any affinity between clients and specific servers.</span></span> <span data-ttu-id="60166-135">Каждый сервер может обрабатывать любой запрос от любого клиента.</span><span class="sxs-lookup"><span data-stu-id="60166-135">Any server can handle any request from any client.</span></span> <span data-ttu-id="60166-136">Тем не менее, другие факторы могут ограничить масштабируемость.</span><span class="sxs-lookup"><span data-stu-id="60166-136">That said, other factors can limit scalability.</span></span> <span data-ttu-id="60166-137">Например, многие веб-службы ведут запись в серверное хранилище данных, которое может быть трудно развернуть. (В статье [Секционирование данных](./data-partitioning.md) описаны стратегии для развертывания хранилища данных.)</span><span class="sxs-lookup"><span data-stu-id="60166-137">For example, many web services write to a backend data store, which may be hard to scale out. - <span data-ttu-id="60166-138">Интерфейсами REST API управляют ссылки на гиперсреды, которые содержатся в представлении.</span><span class="sxs-lookup"><span data-stu-id="60166-138">REST APIs are driven by hypermedia links that are contained in the representation.</span></span> <span data-ttu-id="60166-139">Например, ниже показано JSON-представление заказа.</span><span class="sxs-lookup"><span data-stu-id="60166-139">For example, the following shows a JSON representation of an order.</span></span> <span data-ttu-id="60166-140">Оно содержит ссылки для получения или обновления клиента, связанного с заказом.</span><span class="sxs-lookup"><span data-stu-id="60166-140">It contains links to get or update the customer associated with the order.</span></span>

    ```json
    {
        "orderID":3,
        "productID":2,
        "quantity":4,
        "orderValue":16.60,
        "links": [
            {"rel":"product","href":"https://adventure-works.com/customers/3", "action":"GET" },
            {"rel":"product","href":"https://adventure-works.com/customers/3", "action":"PUT" }
        ]

    ```

<span data-ttu-id="60166-141">В 2008 году Леонард Ричардсон предложил следующую [модель зрелости](https://martinfowler.com/articles/richardsonMaturityModel.html) для веб-API:</span><span class="sxs-lookup"><span data-stu-id="60166-141">In 2008, Leonard Richardson proposed the following [maturity model](https://martinfowler.com/articles/richardsonMaturityModel.html) for web APIs:</span></span>

- <span data-ttu-id="60166-144">Уровень 2. Использование методов HTTP для определения операций с ресурсами.</span><span class="sxs-lookup"><span data-stu-id="60166-144">Level 2: Use HTTP methods to define operations on resources.</span></span>
- <span data-ttu-id="60166-145">Уровень 3. Использование гиперсред (HATEOAS, описан ниже).</span><span class="sxs-lookup"><span data-stu-id="60166-145">Level 3: Use hypermedia (HATEOAS, described below).</span></span>

## <a name="organize-the-api-around-resources"></a><span data-ttu-id="60166-148">Упорядочивание API вокруг ресурсов</span><span class="sxs-lookup"><span data-stu-id="60166-148">Organize the API around resources</span></span>

<span data-ttu-id="60166-149">Сосредоточьтесь на бизнес-сущностях, предоставляемых веб-API.</span><span class="sxs-lookup"><span data-stu-id="60166-149">Focus on the business entities that the web API exposes.</span></span> <span data-ttu-id="60166-150">Например, в системе электронной коммерции основными сущностями могут быть клиенты и заказы.</span><span class="sxs-lookup"><span data-stu-id="60166-150">For example, in an e-commerce system, the primary entities might be customers and orders.</span></span> <span data-ttu-id="60166-151">Создание заказа может осуществляться путем отправки HTTP-запроса POST, содержащего сведения о заказе.</span><span class="sxs-lookup"><span data-stu-id="60166-151">Creating an order can be achieved by sending an HTTP POST request that contains the order information.</span></span> <span data-ttu-id="60166-152">HTTP-ответ указывает на успешность размещения заказа.</span><span class="sxs-lookup"><span data-stu-id="60166-152">The HTTP response indicates whether the order was placed successfully or not.</span></span> <span data-ttu-id="60166-153">Когда это возможно, универсальные коды ресурсов должны основываться на существительных (ресурсе), а не на глаголах (операциях c ресурсом).</span><span class="sxs-lookup"><span data-stu-id="60166-153">When possible, resource URIs should be based on nouns (the resource) and not verbs (the operations on the resource).</span></span>

<span data-ttu-id="60166-154">Ресурс не обязательно должен быть основан на одном физическом элементе данных.</span><span class="sxs-lookup"><span data-stu-id="60166-154">A resource does not have to be based on a single physical data item.</span></span> <span data-ttu-id="60166-155">Например, ресурс заказа может быть реализован внутри системы с использованием нескольких таблиц в реляционной базе данных, однако представлен в клиентском приложении как единая сущность.</span><span class="sxs-lookup"><span data-stu-id="60166-155">For example, an order resource might be implemented internally as several tables in a relational database, but presented to the client as a single entity.</span></span> <span data-ttu-id="60166-156">Избегайте создания API, которые просто отражают внутреннюю структуру базы данных.</span><span class="sxs-lookup"><span data-stu-id="60166-156">Avoid creating APIs that simply mirror the internal structure of a database.</span></span> <span data-ttu-id="60166-157">Цель REST — моделировать сущности и операции, которые приложение может выполнять в этих сущностях.</span><span class="sxs-lookup"><span data-stu-id="60166-157">The purpose of REST is to model entities and the operations that an application can perform on those entities.</span></span> <span data-ttu-id="60166-158">Внутренняя реализация должна быть скрыта от клиента.</span><span class="sxs-lookup"><span data-stu-id="60166-158">A client should not be exposed to the internal implementation.</span></span>

<span data-ttu-id="60166-162">В результате отправки HTTP-запроса GET на URI коллекции возвращается список элементов в коллекции.</span><span class="sxs-lookup"><span data-stu-id="60166-162">Sending an HTTP GET request to the collection URI retrieves a list of items in the collection.</span></span> <span data-ttu-id="60166-163">Каждый элемент в коллекции также имеет свой собственный универсальный код ресурса (URI).</span><span class="sxs-lookup"><span data-stu-id="60166-163">Each item in the collection also has its own unique URI.</span></span> <span data-ttu-id="60166-164">HTTP-запрос GET к URI элемента возвращает подробные сведения об этом элементе.</span><span class="sxs-lookup"><span data-stu-id="60166-164">An HTTP GET request to the item's URI returns the details of that item.</span></span>
