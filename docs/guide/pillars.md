## <a name="scalability"></a><span data-ttu-id="8f02d-117">Масштабируемость</span><span class="sxs-lookup"><span data-stu-id="8f02d-117">Scalability</span></span>

<span data-ttu-id="8f02d-118">Масштабируемость — это способность системы успешно функционировать при повышении нагрузки.</span><span class="sxs-lookup"><span data-stu-id="8f02d-118">Scalability is the ability of a system to handle increased load.</span></span> <span data-ttu-id="8f02d-119">Есть два основных способа масштабирования приложения.</span><span class="sxs-lookup"><span data-stu-id="8f02d-119">There are two main ways that an application can scale.</span></span> <span data-ttu-id="8f02d-120">Вертикальное масштабирование (*увеличение* масштаба) означает увеличение емкости ресурса, например, за счет увеличения размера виртуальной машины.</span><span class="sxs-lookup"><span data-stu-id="8f02d-120">Vertical scaling (scaling *up*) means increasing the capacity of a resource, for example by using a larger VM size.</span></span> <span data-ttu-id="8f02d-121">Горизонтальное масштабирование (*развертывание*) — это добавление новых экземпляров ресурса, например виртуальных машин или реплик базы данных.</span><span class="sxs-lookup"><span data-stu-id="8f02d-121">Horizontal scaling (scaling *out*) is adding new instances of a resource, such as VMs or database replicas.</span></span> 

<span data-ttu-id="8f02d-122">Горизонтальное масштабирование имеет значительные преимущества по сравнению с вертикальным:</span><span class="sxs-lookup"><span data-stu-id="8f02d-122">Horizontal scaling has significant advantages over vertical scaling:</span></span>

- <span data-ttu-id="8f02d-125">Горизонтальное масштабирование является эластичным.</span><span class="sxs-lookup"><span data-stu-id="8f02d-125">Horizontal scale is elastic.</span></span> <span data-ttu-id="8f02d-126">При повышении нагрузки можно добавлять экземпляры, а при ее понижении — удалять их.</span><span class="sxs-lookup"><span data-stu-id="8f02d-126">You can add more instances if load increases, or remove them during quieter periods.</span></span>
- <span data-ttu-id="8f02d-127">Масштабирование может активироваться автоматически, по расписанию или в соответствии с изменением нагрузки.</span><span class="sxs-lookup"><span data-stu-id="8f02d-127">Scaling out can be triggered automatically, either on a schedule or in response to changes in load.</span></span>

- <span data-ttu-id="8f02d-130">Кроме того, горизонтальное масштабирование повышает устойчивость за счет избыточности.</span><span class="sxs-lookup"><span data-stu-id="8f02d-130">Horizontal scaling can also improve resiliency, by adding redundancy.</span></span> <span data-ttu-id="8f02d-131">Если экземпляр выходит из строя, приложение продолжает работать.</span><span class="sxs-lookup"><span data-stu-id="8f02d-131">If an instance goes down, the application keeps running.</span></span>

<span data-ttu-id="8f02d-132">Преимущество вертикального масштабирования состоит в том, что его можно реализовать, не внося изменения в приложение.</span><span class="sxs-lookup"><span data-stu-id="8f02d-132">An advantage of vertical scaling is that you can do it without making any changes to the application.</span></span> <span data-ttu-id="8f02d-133">Но в определенный момент будет достигнут предел и увеличить масштаб больше не удастся.</span><span class="sxs-lookup"><span data-stu-id="8f02d-133">But at some point you'll hit a limit, where you can't scale any up any more.</span></span> <span data-ttu-id="8f02d-134">В таком случае дальнейшее масштабирование должно быть горизонтальным.</span><span class="sxs-lookup"><span data-stu-id="8f02d-134">At that point, any further scaling must be horizontal.</span></span> 

<span data-ttu-id="8f02d-135">Горизонтальное масштабирование необходимо разрабатывать в составе системы.</span><span class="sxs-lookup"><span data-stu-id="8f02d-135">Horizontal scale must be designed into the system.</span></span> <span data-ttu-id="8f02d-136">Например, можно развернуть виртуальные машины, поместив их за подсистемой балансировки нагрузки.</span><span class="sxs-lookup"><span data-stu-id="8f02d-136">For example, you can scale out VMs by placing them behind a load balancer.</span></span> <span data-ttu-id="8f02d-137">Но каждая виртуальная машина в пуле должна быть способна обработать любой клиентский запрос. <span data-ttu-id="8f02d-140">Но для масштабирования недостаточно просто добавить дополнительные экземпляры.</span><span class="sxs-lookup"><span data-stu-id="8f02d-140">Just adding more instances doesn't mean an application will scale, however.</span></span> <span data-ttu-id="8f02d-141">После этого узкое место может попросту образоваться на другом участке.</span><span class="sxs-lookup"><span data-stu-id="8f02d-141">It might simply push the bottleneck somewhere else.</span></span> <span data-ttu-id="8f02d-142">Например, если масштабировать внешний веб-интерфейс для обработки большего количества клиентских запросов, в базе данных могут возникнуть конфликты блокировки.</span><span class="sxs-lookup"><span data-stu-id="8f02d-142">For example, if you scale a web front-end to handle more client requests, that might trigger lock contentions in the database.</span></span> <span data-ttu-id="8f02d-143">В таком случае необходимо учитывать дополнительные показатели, например оптимистическую блокировку или секционирование данных, чтобы повысить пропускную способность при подключении к базе данных.</span><span class="sxs-lookup"><span data-stu-id="8f02d-143">You would then need to consider additional measures, such as optimistic concurrency or data partitioning, to enable more throughput to the database.</span></span>
