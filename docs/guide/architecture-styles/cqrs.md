# <a name="cqrs-architecture-style"></a><span data-ttu-id="3e848-103">Стиль архитектуры CQRS</span><span class="sxs-lookup"><span data-stu-id="3e848-103">CQRS architecture style</span></span>

<span data-ttu-id="3e848-104">Модель CQRS — это стиль архитектуры, в котором операции чтения отделены от операций записи.</span><span class="sxs-lookup"><span data-stu-id="3e848-104">Command and Query Responsibility Segregation (CQRS) is an architecture style that separates read operations from write operations.</span></span> 

![](./images/cqrs-logical.svg)

<span data-ttu-id="3e848-105">В традиционных архитектурах аналогичная модель используется в базах данных для выполнения запросов и обновлений.</span><span class="sxs-lookup"><span data-stu-id="3e848-105">In traditional architectures, the same data model is used to query and update a database.</span></span> <span data-ttu-id="3e848-106">Этот простой подход отлично подходит для базовых операций создания, чтения, обновления и удаления.</span><span class="sxs-lookup"><span data-stu-id="3e848-106">That's simple and works well for basic CRUD operations.</span></span> <span data-ttu-id="3e848-107">Но в более сложных приложениях такой подход не всегда будет удобным.</span><span class="sxs-lookup"><span data-stu-id="3e848-107">In more complex applications, however, this approach can become unwieldy.</span></span> <span data-ttu-id="3e848-108">Например, при чтении данных приложение может выполнять много разных запросов и возвращать несколько объектов передачи данных разных форм.</span><span class="sxs-lookup"><span data-stu-id="3e848-108">For example, on the read side, the application may perform many different queries, returning data transfer objects (DTOs) with different shapes.</span></span> <span data-ttu-id="3e848-109">Это усложняет сопоставление объектов.</span><span class="sxs-lookup"><span data-stu-id="3e848-109">Object mapping can become complicated.</span></span> <span data-ttu-id="3e848-110">Для записи данных модель может применять сложные процедуры проверки и процессы бизнес-логики.</span><span class="sxs-lookup"><span data-stu-id="3e848-110">On the write side, the model may implement complex validation and business logic.</span></span> <span data-ttu-id="3e848-111">В результате вы получите слишком сложную модель, которая выполняет слишком много функций.</span><span class="sxs-lookup"><span data-stu-id="3e848-111">As a result, you can end up with an overly complex model that does too much.</span></span>

<span data-ttu-id="3e848-112">Еще одна возможная проблема заключается в том, что рабочие нагрузки чтения и записи часто асимметричны и имеют совершенно разные требования к производительности и масштабируемости.</span><span class="sxs-lookup"><span data-stu-id="3e848-112">Another potential problem is that read and write workloads are often asymmetrical, with very different performance and scale requirements.</span></span> 

<span data-ttu-id="3e848-113">Модель CQRS устраняет эти проблемы, разделяя потоки чтения и записи: для обновления данных используются **команды**, а для чтения — **запросы**.</span><span class="sxs-lookup"><span data-stu-id="3e848-113">CQRS addresses these problems by separating reads and writes into separate models, using **commands** to update data, and **queries** to read data.</span></span>

- <span data-ttu-id="3e848-114">В основе команд должны быть задачи, а не данные.</span><span class="sxs-lookup"><span data-stu-id="3e848-114">Commands should be task based, rather than data centric.</span></span> <span data-ttu-id="3e848-115">(Например, команда может выполнять функцию "Зарезервировать номер в отеле", но не "Установить значение Reserved для свойства ReservationStatus".) Команды лучше помещать в асинхронную очередь, а не требовать синхронной обработки.</span><span class="sxs-lookup"><span data-stu-id="3e848-115">("Book hotel room," not "set ReservationStatus to Reserved.") Commands may be placed on a queue for asynchronous processing, rather than being processed synchronously.</span></span>

- <span data-ttu-id="3e848-116">Запросы никогда не изменяют данные в базе данных.</span><span class="sxs-lookup"><span data-stu-id="3e848-116">Queries never modify the database.</span></span> <span data-ttu-id="3e848-117">Запрос возвращает объект передачи данных, который не содержит сведения о предметной области.</span><span class="sxs-lookup"><span data-stu-id="3e848-117">A query returns a DTO that does not encapsulate any domain knowledge.</span></span>

<span data-ttu-id="3e848-118">Для дополнительной изоляции можно физически разделить данные для чтения и данные для записи.</span><span class="sxs-lookup"><span data-stu-id="3e848-118">For greater isolation, you can physically separate the read data from the write data.</span></span> <span data-ttu-id="3e848-119">В этом случае в базе данных для чтения можно оптимизировать схему данных так, чтобы максимально эффективно выполнять запросы.</span><span class="sxs-lookup"><span data-stu-id="3e848-119">In that case, the read database can use its own data schema that is optimized for queries.</span></span> <span data-ttu-id="3e848-120">Например, в ней можно хранить [материализованные представления][materialized-view] данных, чтобы не использовать сложные операции соединения или сложные объектно-реляционные сопоставления.</span><span class="sxs-lookup"><span data-stu-id="3e848-120">For example, it can store a [materialized view][materialized-view] of the data, in order to avoid complex joins or complex O/RM mappings.</span></span> <span data-ttu-id="3e848-121">Вы можете даже применить другой тип хранилища данных.</span><span class="sxs-lookup"><span data-stu-id="3e848-121">It might even use a different type of data store.</span></span> <span data-ttu-id="3e848-122">Например, база данных для записи останется реляционной, а для чтения вы примените базу данных документов.</span><span class="sxs-lookup"><span data-stu-id="3e848-122">For example, the write database might be relational, while the read database is a document database.</span></span>

<span data-ttu-id="3e848-123">Если для чтения и записи используются разные базы данных, для них нужно поддерживать синхронизацию. Обычно для этого модель записи публикует событие при каждом обновлении базы данных.</span><span class="sxs-lookup"><span data-stu-id="3e848-123">If separate read and write databases are used, they must be kept in sync. Typically this is accomplished by  having the write model publish an event whenever it updates the database.</span></span> <span data-ttu-id="3e848-124">Обновление базы данных и публикации события должны выполняться в рамках одной транзакции.</span><span class="sxs-lookup"><span data-stu-id="3e848-124">Updating the database and publishing the event must occur in a single transaction.</span></span> 

<span data-ttu-id="3e848-125">Некоторые реализации CQRS используют [шаблон источников событий][event-sourcing].</span><span class="sxs-lookup"><span data-stu-id="3e848-125">Some implementations of CQRS use the [Event Sourcing pattern][event-sourcing].</span></span> <span data-ttu-id="3e848-126">В этом шаблоне состояние приложения сохраняется как последовательность событий.</span><span class="sxs-lookup"><span data-stu-id="3e848-126">With this pattern, application state is stored as a sequence of events.</span></span> <span data-ttu-id="3e848-127">Каждое событие обозначает некоторый набор изменений в данных.</span><span class="sxs-lookup"><span data-stu-id="3e848-127">Each event represents a set of changes to the data.</span></span> <span data-ttu-id="3e848-128">Текущее состояние воссоздается при последовательном воспроизведении событий.</span><span class="sxs-lookup"><span data-stu-id="3e848-128">The current state is constructed by replaying the events.</span></span> <span data-ttu-id="3e848-129">В контексте CQRS шаблон источников событий удобен тем, что эти же события могут оповещать другие компоненты, в частности модель чтения.</span><span class="sxs-lookup"><span data-stu-id="3e848-129">In a CQRS context, one benefit of Event Sourcing is that the same events can be used to notify other components &mdash; in particular, to notify the read model.</span></span> <span data-ttu-id="3e848-130">Модель чтения на основе событий поддерживает моментальный снимок текущего состояния, повышая эффективность запросов.</span><span class="sxs-lookup"><span data-stu-id="3e848-130">The read model uses the events to create a snapshot of the current state, which is more efficient for queries.</span></span> <span data-ttu-id="3e848-131">Но учтите, что шаблон источников событий усложняет архитектуру.</span><span class="sxs-lookup"><span data-stu-id="3e848-131">However, Event Sourcing adds complexity to the design.</span></span>

![](./images/cqrs-events.svg)

## <a name="when-to-use-this-architecture"></a><span data-ttu-id="3e848-132">Когда следует использовать эту архитектуру</span><span class="sxs-lookup"><span data-stu-id="3e848-132">When to use this architecture</span></span>

<span data-ttu-id="3e848-133">Использовать CQRS можно в системах для совместной работы, где большое число пользователей обращается к одним и тем же данным, особенно если рабочие нагрузки чтения и записи явно асимметричны.</span><span class="sxs-lookup"><span data-stu-id="3e848-133">Consider CQRS for collaborative domains where many users access the same data, especially when the read and write workloads are asymmetrical.</span></span>

<span data-ttu-id="3e848-134">CQRS нельзя считать архитектурой верхнего уровня, применимой в масштабе всей системы.</span><span class="sxs-lookup"><span data-stu-id="3e848-134">CQRS is not a top-level architecture that applies to an entire system.</span></span> <span data-ttu-id="3e848-135">Применяйте CQRS только к тем подсистемам, где заметны явные преимущества от разделения процессов чтения и записи.</span><span class="sxs-lookup"><span data-stu-id="3e848-135">Apply CQRS only to those subsystems where there is clear value in separating reads and writes.</span></span> <span data-ttu-id="3e848-136">В противном случае вы только повысите сложность системы, не получая никаких преимуществ.</span><span class="sxs-lookup"><span data-stu-id="3e848-136">Otherwise, you are creating additional complexity for no benefit.</span></span>

## <a name="benefits"></a><span data-ttu-id="3e848-137">Преимущества</span><span class="sxs-lookup"><span data-stu-id="3e848-137">Benefits</span></span>

- <span data-ttu-id="3e848-138">**Независимое масштабирование**.</span><span class="sxs-lookup"><span data-stu-id="3e848-138">**Independently scaling**.</span></span> <span data-ttu-id="3e848-139">CQRS позволяет раздельно масштабировать рабочие нагрузки чтения и записи, снижая риск конфликтов блокировки.</span><span class="sxs-lookup"><span data-stu-id="3e848-139">CQRS allows the read and write workloads to scale independently, and may result in fewer lock contentions.</span></span>
- <span data-ttu-id="3e848-140">**Оптимизация схем данных.**</span><span class="sxs-lookup"><span data-stu-id="3e848-140">**Optimized data schemas.**</span></span>  <span data-ttu-id="3e848-141">Для процессов чтения можно применить схему, оптимизированную для запросов, а для процессов записи — другую схему, оптимизированную для обновлений.</span><span class="sxs-lookup"><span data-stu-id="3e848-141">The read side can use a schema that is optimized for queries, while the write side uses a schema that is optimized for updates.</span></span>  
- <span data-ttu-id="3e848-142">**Безопасность**.</span><span class="sxs-lookup"><span data-stu-id="3e848-142">**Security**.</span></span> <span data-ttu-id="3e848-143">Так будет проще назначить для выполнения операций записи данных только допустимые сущности домена.</span><span class="sxs-lookup"><span data-stu-id="3e848-143">It's easier to ensure that only the right domain entities are performing writes on the data.</span></span>
- <span data-ttu-id="3e848-144">**Четкое разделение зон ответственности**.</span><span class="sxs-lookup"><span data-stu-id="3e848-144">**Separation of concerns**.</span></span> <span data-ttu-id="3e848-145">Разделение процессов чтения и записи позволяет получить более гибкие и простые в обслуживании.</span><span class="sxs-lookup"><span data-stu-id="3e848-145">Segregating the read and write sides can result in models that are more maintainable and flexible.</span></span> <span data-ttu-id="3e848-146">Большая часть сложной бизнес-логики переместится в модель записи.</span><span class="sxs-lookup"><span data-stu-id="3e848-146">Most of the complex business logic goes into the write model.</span></span> <span data-ttu-id="3e848-147">Это в некоторой степени упростит модель чтения.</span><span class="sxs-lookup"><span data-stu-id="3e848-147">The read model can be relatively simple.</span></span>
- <span data-ttu-id="3e848-148">**Более простые запросы**.</span><span class="sxs-lookup"><span data-stu-id="3e848-148">**Simpler queries**.</span></span> <span data-ttu-id="3e848-149">Сохраняя в базе данных для чтения материализованное представление данных, вы предотвратите использование приложением сложных соединений в запросах.</span><span class="sxs-lookup"><span data-stu-id="3e848-149">By storing a materialized view in the read database, the application can avoid complex joins when querying.</span></span>
