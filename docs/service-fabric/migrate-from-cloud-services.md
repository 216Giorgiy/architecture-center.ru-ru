# <a name="migrate-an-azure-cloud-services-application-to-azure-service-fabric"></a>Перенос приложения облачных служб Azure в Azure Service Fabric 

[![GitHub](../_images/github.png) Пример кода][sample-code]

В этой статье описывается перенос приложения из облачных служб Azure в Azure Service Fabric. Основное внимание уделяется архитектурным решениям и рекомендациям по работе. 

Для этого проекта мы начали работу с приложением облачных служб Surveys и перенесли его в Service Fabric. Нам нужно было перенести приложение, сделав при этом как можно меньше изменений. В последующих статьях мы оптимизируем приложение Service Fabric, внедрив архитектуру микрослужб.

Прежде чем ознакомиться с этой статьей, будет полезно понять базовые принципы Service Fabric и архитектуры микрослужб в целом. Ознакомьтесь со следующими статьями:

- [Общие сведения о Service Fabric][sf-overview]
- [Разработка приложений с использованием микрослужб][sf-why-microservices]


## <a name="about-the-surveys-application"></a>О приложении Surveys

В 2012 г. группа по разработке шаблонов и рекомендаций создала приложение Surveys для книги о [разработке многопользовательских приложений для облака][tailspin-book]. В книге описана вымышленная компания Tailspin, проектирующая и внедряющая приложения Surveys.

Surveys является многопользовательским приложением, позволяющим заказчикам создавать опросы. После регистрации в приложении члены организации клиента могут создавать и публиковать опросы, а также собирать результаты для анализа. Приложение включает общедоступный веб-сайт, где пользователи могут принять участие в опросе. Дополнительные сведения об исходном сценарии Tailspin см. [здесь][tailspin-scenario].

Теперь Tailspin хочет переместить приложение Surveys в архитектуру микрослужб с помощью Service Fabric под управлением Azure. Так как приложение уже развернуто в качестве приложения облачных служб, Tailspin реализует многоэтапный подход:

1.  Перенос облачных служб в Service Fabric с минимальными изменениями в приложении.
2.  Оптимизация приложения для Service Fabric путем переноса архитектуры микрослужб.

В этой статье описан первый этап. Второй этап описан в последующих статьях. В реальном проекте вполне вероятно, что оба этапа будут перекрываться. При переносе в Service Fabric приложение перестроится в микрослужбы. Позже вы можете улучшить архитектуру, разделив недетализированные службы на более мелкие.  

Код приложения доступен на [GitHub][sample-code]. Этот репозиторий содержит приложение облачных служб и версию Service Fabric. 

> Облачная служба представляет собой обновленную версию исходного приложения из книги о *разработке многопользовательских приложений*.

## <a name="why-microservices"></a>Зачем нужны микрослужбы

В этой статье нет подробного описания микрослужб. Ниже представлены некоторые из преимуществ, которые компания Tailspin надеется получить, переместив архитектуру микрослужб:

- **Обновление приложений.** Службы можно развернуть независимо друг от друга для поэтапного обновления приложения.
- **Устойчивость и изоляция сбоев.** При сбое в работе службы другие службы продолжают работу.
- **Масштабируемость**. Службы можно масштабировать независимо друг от друга.
- **Гибкость.** Службы разрабатываются на основе бизнес-сценариев, а не стеков технологий, что упрощает перенос служб для использования новых хранилищ данных, платформ или технологий.
- **Гибкая разработка.** В отдельных службах меньше кода, чем в монолитных приложениях. Благодаря этому база кода проще для понимания, анализа и тестирования.
- **Небольшие специализированные команды.** Так как приложение разбито на много небольших служб, разработкой каждой из них может заниматься небольшая специализированная команда.

## <a name="why-service-fabric"></a>Выбор Service Fabric
      
Service Fabric является удачным выбором для архитектуры микрослужб, так как большая часть функций, необходимых в распределенной системе, встроены в Service Fabric, включая следующие:

- **Управление кластерами.** Service Fabric автоматически выполняет отработку отказа узла, мониторинг работоспособности и другие функции управления кластером.
- **Горизонтальное масштабирование.** При добавлении узлов в кластер Service Fabric приложение автоматически масштабируется по мере распределения служб на новых узлах.
- **Обнаружение служб.** Service Fabric предоставляет службу обнаружения, которая может разрешать конечную точку для конкретной службы.
- **Службы без отслеживания состояния и с отслеживанием состояния.** Службы с отслеживанием состояния используют [надежные коллекции][sf-reliable-collections], которые можно использовать вместо кэша или очереди, а также разделить.
- **Управление жизненным циклом приложений.** Службы могут обновляться независимо друг от друга без простоя приложений.
- **Оркестрация служб** в кластере компьютеров.
- **Более высокая плотность** для оптимизации потребления ресурсов. Несколько служб могут размещаться на одном узле.

Service Fabric используется различными службами Майкрософт, включая базу данных SQL Azure, Cosmos DB, концентраторы событий Azure и т. д., благодаря чему это проверенная платформа для создания распределенных облачных приложений. 

## <a name="comparing-cloud-services-with-service-fabric"></a>Сравнение облачных служб и Service Fabric

В следующей таблице перечислены некоторые важные различия между приложениями облачных служб и Service Fabric. Более подробные сведения см. в статье [о различиях между облачными службами и Service Fabric][sf-compare-cloud-services].

|        | Облачные службы | Service Fabric |
|--------|---------------|----------------|
| Структура приложения | Роли| Службы |
| Плотность |Один экземпляр роли на каждой виртуальной машине | Несколько служб на одном узле |
| Минимальное количество узлов | 2 на роль | 5 на кластер, в рабочей среде |
| Управление данными о состоянии | Без отслеживания состояния | Без отслеживания или с отслеживанием состояния* |
| Hosting | Таблицы Azure | Облачные или локальные системы. |
| Размещение веб-сайта | IIS* | Автономное размещение |
| Модель развертывания | [Классическая модель развертывания][azure-deployment-models] | [Resource Manager][azure-deployment-models]  |
| Упаковка | Файлы пакета облачной службы (CSPKG) | Пакеты приложения и служб |
| Обновление приложения | Переключение виртуальных IP-адресов или последовательное обновление | Последовательное обновление |
| Автомасштабирование | [Встроенная служба][cloud-service-autoscale] | Масштабируемые наборы виртуальных машин для автоматического развертывания |
| Отладка | Локальный эмулятор | Локальный кластер |


\*Службы с отслеживанием состояния используют [надежные коллекции][sf-reliable-collections] для сохранения состояния между репликами, чтобы все операции выполнялись локально на узлах в кластере. Операции записи реплицируются по узлам для обеспечения надежности. Для служб без отслеживания состояния может быть предусмотрено внешнее состояние с использованием базы данных или других внешних хранилищ.

** Рабочие роли также могут самостоятельно размещать веб-API ASP.NET с помощью OWIN.

## <a name="the-surveys-application-on-cloud-services"></a>Приложение Surveys в облачных службах

На следующей схеме показана архитектура приложения Surveys в облачных службах. 

![](./images/tailspin01.png)

Приложение состоит из двух веб-ролей и рабочей роли.

- Веб-роль **Tailspin.Web** размещает веб-узел ASP.NET, который клиенты Tailspin используют для создания опросов и управления ими. Клиенты также используют этот веб-сайт для регистрации в приложении и управления своими подписками. Администраторы Tailspin могут использовать его для просмотра списка клиентов и управления их данными. 

- Веб-роль **Tailspin.Web.Survey.Public** размещает веб-сайт ASP.NET, где пользователи могут участвовать в опросах, опубликованных клиентами Tailspin. 

- Рабочая роль **Tailspin.Workers.Survey** выполняет фоновую обработку. Веб-роли помещают рабочие элементы в очередь, а рабочая роль обрабатывает их. Определяются две фоновые задачи: экспорт ответов на опрос в базу данных SQL Azure и расчет статистики для них.

Помимо облачных служб приложение Surveys использует и другие службы Azure:

- **Службу хранилища Azure** для хранения опросов, ответов на опросы и информации о клиенте.

- **Кэш Redis для Azure** для кэширования данных, хранящихся в службе хранилища Azure, чтобы обеспечить быстрый доступ на чтение. 

- **Azure Active Directory** (Azure AD) для проверки подлинности клиентов и администраторов Tailspin.

- **Базу данных SQL Azure** для хранения ответов на опросы для анализа.

### <a name="creating-the-web-front-end-services"></a>Создание службы веб-интерфейса

В Service Fabric служба выполняется внутри процесса, созданного средой выполнения Service Fabric. Для веб-интерфейса это означает, что служба не выполняется в IIS. Вместо этого в службе должен размещаться веб-сервер. Такой подход называется *автономным размещением*, так как код, который выполняется внутри процесса, выступает в качестве хоста веб-сервера. 

Требования к автономному размещению означает, что служба Service Fabric не может использовать ASP.NET MVC или веб-формы ASP.NET, так как этим платформам требуются службы IIS и они не поддерживают автономное размещение. Далее представлены варианты автономного размещения.

- [ASP.NET Core][aspnet-core] — автономное размещение с помощью веб-сервера [Kestrel][kestrel]. 
- [Веб-API ASP.NET][aspnet-webapi] — автономное размещение с помощью [OWIN][owin].
- Сторонние платформы, такие как [Nancy](http://nancyfx.org/).

В исходном приложении Surveys используется ASP.NET MVC. Так как ASP.NET MVC не может автономно размещаться в Service Fabric, мы рассматриваем следующие варианты миграции:

- Перенос веб-роли в ASP.NET Core, которая может быть размещена автономно.
- Преобразование веб-сайта в приложение с одной страницей (SPA), которое вызывает веб-API, реализованный с помощью веб-API ASP.NET. Это потребовало бы полной переработки веб-интерфейса.
- Сохранение имеющегося кода ASP.NET MVC и развертывание служб IIS в контейнере Windows Server в Service Fabric. Такой подход потребует минимум изменений кода или не потребует их вовсе. Тем не менее [поддержка контейнеров][sf-containers] в Service Fabric сейчас находится на стадии предварительной версии.

Исходя из указанного, мы выбрали первый вариант — перенос в ASP.NET Core. Для этого мы выполнили инструкции из статьи [Migrating From ASP.NET MVC to ASP.NET Core MVC][aspnet-migration] (Миграция из ASP.NET MVC в ASP.NET Core MVC). 

> [!NOTE]
> По соображениям безопасности при использовании ASP.NET Core с Kestrel необходимо поместить перед Kestrel обратный прокси-сервер для обработки трафика из Интернета. Дополнительные сведения см. в статье [Kestrel web server implementation in ASP.NET Core][kestrel] (Реализация веб-сервера Kestrel в ASP.NET Core). В разделе [Развертывание приложения](#deploying-the-application) описывается рекомендуемое развертывание Azure.

### <a name="http-listeners"></a>HTTP-прослушиватели

В облачных службах рабочие и веб-роли предоставляют конечную точку HTTP, объявив ее в [файле определения службы][cloud-service-endpoints]. Веб-роль должна иметь по крайней мере одну конечную точку.

```xml
<!-- Cloud service endpoint -->
<Endpoints>
    <InputEndpoint name="HttpIn" protocol="http" port="80" />
</Endpoints>
```

Аналогично конечные точки службы Service Fabric объявляются в манифесте службы: 

```xml
<!-- Service Fabric endpoint -->
<Endpoints>
    <Endpoint Protocol="http" Name="ServiceEndpoint" Type="Input" Port="8002" />
</Endpoints>
```

В отличие от роли облачной службы Service Fabric могут располагаться на одном узле. Таким образом, каждая служба должна ожидать передачи данных на отдельном порте. Далее в этой статье мы обсудим, как клиентские запросы на порт 80 или 443 перенаправляются на правильный порт для службы.

Службе необходимо явно создать прослушиватели для каждой конечной точки. Это нужно сделать, так как Service Fabric не зависит от стеков связи. Дополнительные сведения см. в статье [Создание внешнего интерфейса веб-службы для приложения с помощью ASP.NET Core][sf-aspnet-core].

## <a name="packaging-and-configuration"></a>Упаковка и конфигурация

 Облачная служба содержит следующие файлы конфигурации и пакета:

| Файл | ОПИСАНИЕ |
|------|-------------|
| Определение службы (CSDEF) | Параметры, используемые Azure для настройки облачной службы. Определяет роли, конечные точки, задачи запуска и имена параметров конфигурации. |
| Конфигурация службы (CSCFG) | Параметры развертывания, включающие число экземпляров роли, номера портов конечной точки и значения параметров конфигурации. 
| Пакет службы (CSPKG) | Содержит код и конфигурации приложений, а также файл определения службы.  |

Имеется один CSDEF-файл для всего приложения. У вас может быть несколько CSCFG-файлов для разных сред, таких как локальная, рабочая или тестовая. Если служба запущена, можно обновить CSCFG-файл, но не CSDEF-файл. Дополнительные сведения см. в статье [Что такое модель облачных служб и как создать ее пакет?][cloud-service-config]

Service Fabric имеет аналогичное разделение между *определением* и *параметрами* службы, но структура более детальна. Чтобы понять модель конфигурации Service Fabric, важно знать, как упаковано приложение Service Fabric. Ниже приведена структура:

```
Application package
  - Service packages
    - Code package
    - Configuration package
    - Data package (optional)
```

Пакет приложения — это то, что развертывается. Он содержит один или несколько пакетов службы. Пакет службы содержит пакеты кода, конфигурации и данных. Пакет кода содержит двоичные файлы для служб, а пакет конфигурации — параметры конфигурации. Эта модель позволяет обновить отдельные службы без повторного развертывания всего приложения, а также только параметры конфигурации без повторного развертывания кода или перезапуска службы.

Приложение Service Fabric содержит следующие файлы конфигурации:

| Файл | Расположение | ОПИСАНИЕ |
|------|----------|-------------|
| ApplicationManifest.xml | Пакет приложения | Определяет службы, входящие в состав приложения. |
| ServiceManifest.xml | Пакет службы| Описывает одну или несколько служб. |
| Settings.xml | Пакет конфигурации | Содержит параметры конфигурации для службы, определенной в пакете службы. |

Дополнительные сведения см. в статье [Моделирование приложения в структуре службы][sf-application-model].

Для поддержки разных параметров конфигурации для нескольких сред используйте следующий подход, описанный в статье [Управление параметрами приложения для нескольких сред][sf-multiple-environments]:

1. Задайте параметры в файле Setting.xml для службы.
2. В манифесте приложения определите переопределение для параметра.
3. Поместите параметры, относящиеся к среде, в файлы параметров приложения.


## <a name="deploying-the-application"></a>Развертывание приложения

Облачные службы Azure — это управляемая служба, а Service Fabric — среда выполнения. Кластеры Service Fabric можно создавать во многих средах, включая Azure и локальную среду. В этой статье рассматривается развертывание в Azure. 

На следующей схеме показан рекомендуемый тип развертывания:

![](./images/tailspin-cluster.png)

Кластер Service Fabric развертывается в [масштабируемом наборе виртуальных машин][vm-scale-sets]. Масштабируемые наборы относятся к вычислительным ресурсам Azure, которые можно использовать для развертывания набора идентичных виртуальных машин и управления им. 

Как уже упоминалось, по соображениям безопасности веб-серверу Kestrel требуется обратный прокси-сервер. На этой схеме показан [шлюз приложений Azure][application-gateway], который является службой Azure с различными функциями балансировки нагрузки уровня 7. Он выступает в качестве службы обратного прокси-сервера, завершая подключение клиента и перенаправляя запросы к конечным точкам серверной части. Вы можете использовать другое решение обратного прокси-сервера, например nginx.  

### <a name="layer-7-routing"></a>Маршрутизация уровня 7

В [исходном приложении Surveys](https://msdn.microsoft.com/library/hh534477.aspx#sec21) одна веб-роль ожидала передачи данных на порте 80, а другая — на порте 443. 

| Общий сайт | Сайт управления опросом |
|-------------|------------------------|
| `http://tailspin.cloudapp.net` | `https://tailspin.cloudapp.net` |

Кроме того, можно использовать маршрутизацию уровня 7. При таком подходе различные пути URL-адресов будут направляться к разным номерам портов на серверной части. Например, общедоступный сайт может использовать путь URL-адреса, начинающийся с `/public/`. 

Далее представлены варианты для маршрутизации уровня 7.

- Использование шлюза приложений. 

- Используйте сетевое виртуальное устройство, например nginx.

- Запись пользовательского шлюза в качестве службы без отслеживания состояния.

Рассмотрите этот подход, если имеется две или больше служб с общедоступными конечными точками HTTP, но они должны отображаться как один сайт с одним доменным именем.

> Мы *не* рекомендуем разрешать внешним клиентам отправлять запросы через [обратный прокси-сервер][sf-reverse-proxy] Service Fabric. Несмотря на то что это можно сделать, обратный прокси-сервер предназначен для обмена данными между службами. Открыв его для внешних клиентов, вы предоставляете доступ к *любой* службе в кластере с конечной точкой HTTP.

### <a name="node-types-and-placement-constraints"></a>Типы узлов и ограничения по размещению

В развертывании выше все службы работают на всех узлах. Тем не менее службы можно также сгруппировать для запуска определенных служб только на отдельных узлах кластера. Причины применения этого подхода:

- Выполнение некоторых служб на виртуальных машинах разных типов. Например, некоторые службы могут интенсивно использовать вычислительные ресурсы или требовать GPU. В кластере Service Fabric можно использовать виртуальные машины разных типов.
- Изоляция интерфейсных и серверных служб по соображениям безопасности. Все интерфейсные службы будут выполняться в одном наборе узлов, а серверные — в другом в одном кластере.
- Различные требования к масштабированию. Некоторые службы может потребоваться запустить на большем числе узлов, чем другие. Например, при определении интерфейсных и серверных узлов каждый набор может масштабироваться автономно.

На следующей схеме показан кластер, в котором разделены интерфейсные и серверные службы:

![](././images/node-placement.png)

Для реализации такого подхода сделайте следующее:

1.  Во время создания кластера определите два или больше типов узлов. 
2.  Для каждой службы используйте [ограничения на размещение][sf-placement-constraints], чтобы присвоить службу для типа узла.

При развертывании в Azure каждый тип узла развертывается в отдельном масштабируемом наборе виртуальных машин. Кластер Service Fabric обслуживает все типы узлов. Дополнительные сведения см. в статье [Типы узлов Azure Service Fabric и масштабируемые наборы виртуальных машин][sf-node-types].

> Если в кластере имеется несколько типов узлов, один из них используется в качестве *основного*. Службы среды выполнения Service Fabric, например Cluster Management Service, выполняются на основном узле. Подготовьте по крайней мере 5 узлов для основного узла в рабочей среде. Для остальных типов нужно подготовить не менее двух узлов.

## <a name="configuring-and-managing-the-cluster"></a>Настройка кластера и управление им

Кластеры должны быть защищены для предотвращения несанкционированного подключения к ним. Мы рекомендуем использовать Azure AD для проверки подлинности клиентов и сертификаты X.509 для защиты обмена данными между узлами. Дополнительные сведения см. в статье [Сценарии защиты кластера Service Fabric][sf-security].

Сведения о настройке общедоступной конечной точки HTTPS см. в статье [Указание ресурсов в манифесте службы][sf-manifest-resources].

Приложение можно развернуть, добавив виртуальные машины в кластер. Масштабируемые наборы виртуальных машин поддерживают автомасштабирование с помощью правил автомасштабирования на основе счетчиков производительности. Дополнительные сведения см. в статье [Масштабирование кластера Service Fabric с помощью правил автомасштабирования][sf-auto-scale].

Во время работы кластера следует собирать журналы со всех узлов в центральном расположении. Дополнительные сведения см. в статье [Агрегирование и сбор событий с помощью системы диагностики Microsoft Azure][sf-logs].

Кроме того, развертывание изменено с облачных служб на кластер Service Fabric, работающий в масштабируемом наборе виртуальных машин.
